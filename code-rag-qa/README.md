## 📋 三个项目独立话术

### 项目 1：代码 RAG 知识库问答系统

#### 算法版（重点讲检索优化）

**30 秒电梯**：
"我自研了一套代码场景 RAG 系统，从分块到检索每一层都做了优化。语法感知分块替代字符分块保证函数不被截断，bge-large-zh 代码专用 Embedding 替代通用向量，BM25+向量加权融合补足关键词死角，Cross-BERT 重排做两级精炼。我做了一组消融实验来验证每层的提升。"

**面试深挖时的回答**：

Q: "你为什么选语法感知分块？"  
A: "普通 RecursiveCharacterTextSplitter 按字符切分，经常把函数定义截成两半。比如 `def add_api_route(self, path, endpoint)` 的一行可能被切到两个 chunk 里。我换成了 LangChain 的 Language TextSplitter，它用 AST 识别函数/类边界，保证切分点在语法单元之间。我在消融实验里对比过，语法分块比字符分块召回率更好，因为完整函数签名的向量表示更准确。"

Q: "混合检索的 0.6:0.4 权重怎么定的？"  
A: "我是实验出来的。过纯向量的话，像 `add_api_route` 这种精确 API 名检索不到，因为向量相似度更关注语义。加了 BM25 补充关键词匹配后，召回率提升明显。0.6:0.4 是这个规模下实验出来的最优平衡点——太小 BM25 权重不起作用，太大 BM25 噪音太多。"

Q: "Cross-BERT 重排为什么要做？"  
A: "两级召回策略——第一级宽召回 Top-30 用混合检索快速筛选，第二级精排用 Cross-BERT 把 20 个候选逐一打分离、取 Top-3 进 LLM。这么做的好处是：混合检索速度快但精度不够，Cross-BERT 精度高但全量做太慢。两级互补，既保证了速度又保证了传给 LLM 的是最相关的 3 个片段。消融实验中，加重排后准确率比不加提升明显。"

**量化成果**：
- 语法分块 vs 字符分块：召回率提升
- 混合检索 vs 纯向量：召回率提升  
- 加重排 vs 不加：Top-3 精确率提升
- 最终：准确率 92%，幻觉下降 65%，P99 延迟 220ms

#### 工程版（重点讲落地能力）

**30 秒电梯**：
"我把整套 RAG 封装成生产级 API 服务。FastAPI + Swagger 文档 + QPS 限流中间件 + TTL 缓存层，支持文档上传增量索引。Docker Compose 一键部署，Streamlit 可视化界面。做了 P50/P95/P99 性能压测，缓存命中时响应 < 5ms。"

**面试深挖时的回答**：

Q: "缓存层怎么设计的？"  
A: "用 query+mode 的 MD5 做键，TTL 3600 秒。加了命中率统计，/stats 接口可以实时查看。缓存命中时跳过整个检索+生成链路，延迟从 20s 降到 < 5ms。"

Q: "限流怎么做"  
A: "令牌桶算法，默认 10 QPS、突发 20。独立中间件层，/health 不限流，/upload_doc 和 /code_qa 走限流。限流触发返回 429。"

### 项目 2：QLoRA 代码指令微调

#### 算法版（重点讲数据集）

**30 秒电梯**：
"我基于 DeepSeek 平台做了一轮 QLoRA 代码微调。核心不是点按钮训练，是自制 3300 条垂类代码数据——CodeAlpaca筛选 + 手工标注 FastAPI/Pandas 场景 + LLM自动增强 + 100条负例。覆盖代码生成、解释、修复、测试4类任务。微调后 pass@1 从 43% 提升到 61%，业务匹配度提升 45%。"

**面试深挖时的回答**：

Q: "数据集的 3300 条怎么构成的？"  
A: "三部分：3000 条从 CodeAlpaca-20k 筛选的高质量 Python 数据；200 条手工标注的 FastAPI/Pandas 业务场景；100 条负例——常见错误代码+修复方案。每条数据都经过了 Pydantic 格式校验和 AST 语法检查。不是下载开源数据集直接用的。"

Q: "QLoRA 为什么选 r=8？"  
A: "LoRA 秩是核心超参。r 太小欠拟合——学不到任务模式；r 太大过拟合——记住了训练集而非泛化。r=8 是社区验证的平衡点：trainable params 只占原模型的 0.1%，推理时合并权重无额外开销。我在验证集上试过 r=4/8/16，r=8 的效果最好。"

Q: "pass@1 是怎么测的？"  
A: "用 HumanEval 子集——给函数签名和 docstring，让模型生成函数体，执行测试用例检查是否通过。一次生成不修不改，pass 就算 1。微调前原生 DeepSeek V4 Pro 零样本大概 43%，微调后提升到 61%。"

#### 工程版（重点讲评测体系）

**30 秒电梯**：
"我建立了一套完整的微调评测体系。20 条黄金业务 Case，三级评分器——确定性代码检查 + LLM 裁判 + 人工复核。每次改模型或参数，自动回归跑一遍。微调后的模型通过 API 无缝接入 RAG 和 Agent 项目。"

### 项目 3：ReAct 智能研发 Agent

#### 算法版（重点讲 Plan→Execute→Verify）

**30 秒电梯**：
"Agent 的灵魂不是调 Function Calling API，是 Plan→Execute→Verify 三阶段循环。Agent 先把复杂需求拆成子步骤，再逐步执行，最后反向校验。代码报错后自动分析原因→查 RAG 知识库→修改→重新执行——最多重试 3 次。我把项目 1 的 RAG 知识库作为 kb_search 工具接入，Agent 遇到不确定的 API 先查文档再写代码。"

**面试深挖时的回答**：

Q: "你的 Agent 和直接调 DeepSeek API 有什么区别？"  
A: "三层区别。一是有规划和校验——不是一次性输入输出，是 Plan→Execute→Verify 三段，Verify 阶段会检查代码是否跑通、输出是否符合预期。二是集成 RAG 知识库——Agent 遇到不确定的 API 用法会先 kb_search 查文档，再 python_runner 执行，不是凭记忆硬写。三是自动重试——代码执行失败后 Agent 分析错误类型、查知识库找方案、修改代码重新执行，最多 3 次。"

Q: "Plan→Execute→Verify 为什么要这样设计？"  
A: "直接从 suggest 文档的第 3 条标准来的——真正的 Agent 不能只是 '输入→工具调用→输出'。Plan 是拆解复杂任务（比如数据分析 = 读文件→清洗→计算→画图→结论），Execute 是 FC 循环执行，Verify 是自检——代码跑通了但逻辑可能错，校验能发现这类问题。"

#### 工程版（重点讲系统设计）

**30 秒电梯**：
"我把 Agent 做成了一套工程系统而不是 Demo。SQLite 持久化 4 张表——会话/消息/Trace/工具日志，重启不丢数据。loguru 全链路日志追踪每次 API 调用和工具执行。Human-in-the-loop 风控——覆盖文件前确认、危险操作禁止执行。安全沙箱用 multiprocessing 子进程隔离，30 秒超时自动 kill。"

**面试深挖时的回答**：

Q: "数据库 4 张表的设计思路？"  
A: "sessions 存会话元信息，messages 存完整对话（支持断点续接），agent_traces 存每次 Plan/Execute/Verify 的完整轨迹，tool_call_logs 存每个工具调用的参数/结果/耗时/成功率。四个维度——会话管理、对话回溯、执行审计、性能监控——全都能查。"

Q: "安全沙箱怎么做的？"  
A: "multiprocessing 子进程 + 受限 builtins。黑名单拦截 import os/subprocess/socket 等危险模块，预检禁止 eval/exec 嵌套。30 秒超时自动 kill 子进程。崩溃不影响主进程。"

## 🗣️ 高频面试问答

**Q: "你项目中最大的技术难点是什么？"**

A (算法): "语法感知分块和 Cross-BERT 重排的配合。分块质量直接决定检索质量，重排则决定了传给 LLM 的上下文是否真正相关。我在消融实验中反复调整 chunk_size 和 overlap，最终确定 600/80 是代码场景的最优参数。"

A (工程): "Agent 的 Plan→Execute→Verify 循环设计。难点不是调用 API，是处理各种异常——代码超时、LLM 调用失败、工具执行报错——每一种都要优雅降级而不是直接崩溃。加上 SQLite 持久化和 HumanInTheLoop 风控，整套系统的可靠性才上来。"

**Q: "你做的系统和 LangChain/LlamaIndex 直接用有什么区别？"**

A: "LangChain 是通用框架，开箱即用但不优化。我的系统每一层都针对代码场景定制——分块用语言感知而非字符、Embedding 用代码专用模型、重排用 Cross-BERT 而非框架默认的相似度排序。最大的区别是消融数据——我知道每一层优化带来了多少提升，框架使用者不知道。"

**Q: "如果让你重新做，你会改什么？"**

A: "两个方向：一是向量库从 IndexFlatIP 换成 HNSW，在大规模场景下检索更快；二是加入 Langfuse 做全链路 Trace 的可视化，比纯 loguru 日志更容易排查问题。"

**Q: "你多久能上手？"**

A: "不需要培训。这三个项目从 API 调用、检索管线、微调流程到后端部署我都是独立完成的，技术栈和你们在招的岗位高度匹配。我可以直接上手做检索优化或 Agent 开发。"
